# 博客文章配图工作流

这个 skill 用于当前仓库的文章配图工作流。

它既支持首次生成新图，也支持在现有文章配图基础上做重做、补图和风格迭代。

必须使用的 companion skill：

- `imagegen`

默认顺序：

1. 先阅读文章并完成信息结构拆解与版式规划。
2. 借助 Excalidraw 思路辅助梳理结构，也只能停留在内部规划层，不能生成、保存或交付 `.excalidraw` 文件，更不能生成 `.svg` 作为中间产物。
3. 直接用 `imagegen` 生成最终 PNG 成图。

`excalidraw-diagram-generator` 不是必选项，也不是默认执行链路。
这里需要的是 “Excalidraw 式版式语言”，不是 “Excalidraw 文件工作流”。

## 强约束

以下规则是强约束，不满足任意一条都必须停止生成，并明确提示用户原因：

- 最终成品必须由 `imagegen` 直接生成
- 最终交付物必须是 PNG
- 禁止输出 `.svg`
- 禁止输出 `.excalidraw`
- 禁止把 `.excalidraw -> .svg -> .png` 作为生成链路
- 禁止用脚本绘制、拼装或导出“替代 imagegen 的最终成图”
- 如果当前环境无法完成 `imagegen -> PNG` 这条链路，必须停止并告知用户

## 先读什么

开始生成前，先读取：

- 目标文章：`content/posts/*.md`
- 图片规范：`references/image-style.md`

其中 `references/image-style.md` 是这个 skill 的视觉规范源文件。除非用户明确要求改变风格，否则不要绕开或覆盖其中的约束。

## 产物范围

默认只产出：

- 最终 PNG 配图

默认输出目录：

- `public/images/posts/<slug>/`

默认不生成任何中间图源。
即使用户要求“保留图源”或“需要可编辑结构稿”，本 skill 也不输出 `.excalidraw` 或 `.svg`；需要先停下来提示用户当前 skill 的强约束不支持该请求。

## 支持的生成范围

这个 skill 同时支持两类任务：

- 整篇文章配图
- 某一篇文章中的单个小节、单段内容或单个观点配图

如果用户只指定局部内容，则只围绕该范围生成图片，不自动扩展到整篇文章。

## 工作流

### 1. 理解文章

阅读文章后先判断：

- 需要几张图
- 每张图对应哪一段或哪一节
- 每张图只表达哪一个核心观点

风格参考规则：

- 默认优先参考 `references/image-style.md`，并结合目标文章内容完成画面表达
- 不得因为仓库里存在其他文章的配图，就默认继承它们的“系列感”
- 跨文章生成时，除非用户明确要求参考某篇文章，否则不得把别的文章图片当作默认风格来源

如果文章正文已经引用了图片文件名，优先沿用原文件名；除非用户要求改命名。

如果本次任务是重做某张已有图片：

- 先以 `references/image-style.md` 为第一风格来源
- 重做时可以调整构图、文案密度、卡片分组、配色轻重和信息取舍
- 但仍应保持这张图表达的核心观点不偏离原文章对应段落或小节

如果用户指定了某个小节、段落、标题或一句核心观点：

- 只提取该局部内容作为本次配图输入
- 需要时可以参考上下文帮助理解
- 但最终图片只表达该局部的单一核心点

### 2. 先做结构规划

这一阶段只解决“信息怎么排”，不生成任何中间文件：

- 这张图更适合概念卡片、对比图、总结图，还是流程图
- 模块应该如何分组
- 内容更适合 `16:9` 横版，还是适合更自由的纵向/横向长结构
- 是否真的需要 icon
- 可以借鉴 Excalidraw 常见的信息卡片、关系分组、流程拆解和留白组织方式
- 但这种借鉴只能作为内部分析方法，不能变成实际文件产出链路

如果代理发现自己需要依赖 `.excalidraw`、`.svg`、脚本绘图或其他中间导出文件来推进，就必须停止并提示用户，而不是绕路继续生成。

### 3. 再生成最终图片

必须使用 `imagegen` 生成最终 PNG。

提示词要求：

- 遵循 `references/image-style.md`
- 保持“Excalidraw 式版式 + 产品图解式字体”的统一方向
- 优先保证结构清晰、留白充足、字号安全
- 如果内容过多，优先删减文案、拆句、拆图，不要强行缩小字号

在调用 `imagegen` 前，代理应先自检：

- 是否准备直接产出 PNG
- 是否没有引入 `.svg` / `.excalidraw` 中间产物
- 是否没有改用脚本绘图替代 `imagegen`

只要有一项不满足，就必须停止并提示用户。

### 4. 保存与版本管理

输出保存到：

- `public/images/posts/<slug>/`

命名规则：

- 优先复用文章中已经引用的文件名
- 如果是对比版本，可使用 `-v2`、`-16x9`、`-no-icon`、`-font-hierarchy` 等后缀
- 除非用户明确要求替换，否则不要覆盖已有图片

禁止保存以下产物：

- `.svg`
- `.excalidraw`
- 任何仅作为 `imagegen` 替代品的中间导出文件

## 默认项目约束

除非用户另行说明，否则使用以下默认规则：

- 默认不使用 icon
- 默认优先考虑 `16:9`
- 但对于长流程、多层级结构、节点较多的内容，不强制 `16:9`
- 一张图只表达一个核心观点
- 默认追求 “Excalidraw 式结构清晰度”，而不是产出 Excalidraw 文件

## 何时读取 references

以下情况必须读取 `references/image-style.md`：

- 新建文章配图
- 迭代已有图片风格
- 用户提到字体、留白、icon、比例、信息密度、Excalidraw 风格等视觉要求

## 为什么这样组织

这个 skill 采用：

- `SKILL.md` 放工作流与决策规则
- `references/image-style.md` 放详细视觉规范

这样做的好处是：

- 主流程更短，更容易触发和维护
- 视觉规范可以持续迭代，不会让主 skill 变得臃肿
- 避免把同一套要求重复写在多个地方
- 保持 `imagegen -> PNG` 这条最终成图链路单一而明确

## 完成时的输出要求

结束时需要说明：

- 生成或更新了哪些图片
- 保存路径是什么
- 是替换版本还是对比版本
- 如果这次没有使用 `16:9` 或使用了 icon，原因是什么
- 如果因为强约束不满足而停止，也要明确说明是哪一条约束阻止了生成
